# Implementation Plan: Bulletproof Resource-Constrained ETL & AI Pipeline

**Branch**: `007-bulletproof-etl-pipeline` | **Date**: 2026-06-29 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/007-bulletproof-etl-pipeline/spec.md`

## Summary

Spec 006 introduced the deterministic pre-filter (`isRelevantJob`), the cross-training wildcard, a `getFilterProfile()` reader, and a null-summary guardrail. Production logs since 006 reveal the hardening is **incomplete at the seams**: `getFilterProfile()` swallows JSON parse failures into a silent `{}` (no warning, filter silently bypassed); `callOllama()` calls bare `JSON.parse()` on raw model output and throws on truncation/markdown; the ETL loads all ~1,500 jobs into a single in-memory array and never persists a fallback record when the model fails (the item is dropped, not flagged); and first-person inversion is requested-against in the prompt but never *verified* post-generation.

This plan closes those four seams:

1. **Layer 1 вҖ” Preference resolution**: make `getFilterProfile()` casing-tolerant and *loud* вҖ” distinguish "no preferences configured" from "preferences malformed", log an explicit warning on the latter, and apply a safe default instead of an empty object.
2. **Layer 2 вҖ” Memory-bounded batching**: stream scraper output into bounded chunks (default 50) and process each chunk before fetching the next; keep the existing `pLimit(1)` Ollama gate; add a defensive `--max-old-space-size` note.
3. **Layer 3 вҖ” LLM strategy & output integrity**: keep `qwen2.5:0.5b` local as the default (research justifies vs. `llama3.2:1b` and external endpoints), tighten the token-conservation prompt, and add a deterministic first-person inversion detector that swaps inverted summaries for a fallback.
4. **Layer 4 вҖ” Defensive parsing & persistence fallbacks**: insert a stateless `repairAndParse()` stage between the model response and `JSON.parse`, and guarantee every job вҖ” including total Ollama failure вҖ” persists a complete, constraint-valid `ai_analysis` row flagged for review (match_score sentinel + placeholder summary).

## Technical Context

**Language/Version**: TypeScript 5.x, Node.js 20, ESM (`"type": "module"`)

**Primary Dependencies**: oracledb (Thin Mode), Fastify, node-cron, pino, p-limit, Ollama HTTP API

**Storage**: Oracle Autonomous DB (Thin Mode) вҖ” tables `jobs`, `ai_analysis`, `user_profile`, `raw_jobs`. **No schema changes.**

**Testing**: `tsc --noEmit` + Vitest unit tests (existing `*.test.ts` pattern) + manual ETL log inspection

**Target Platform**: Oracle VPS, 1 GB RAM (shared by Fastify API + oracledb thin client + Ollama)

**Project Type**: web-service (backend ETL + REST API)

**Performance Goals**: Full ~1,500-item batch completes within the 3-hour cron window; zero OOM kills; ~80% of batch discarded before any Ollama call

**Constraints**: Oracle Thin Mode only (Principle III). `num_predict: 400` cap. Model fits вүӨ ~700 MB resident alongside Node. No DB schema migration. Summary column rejects NULL/empty (CLOB-empty-as-NULL).

**Scale/Scope**: ~1,500 scraped jobs/run; pre-filter expected to pass ~50вҖ“300 to Ollama; chunk size 50.

### Pipeline State Transitions (per job)

| State | Trigger | Next State |
|---|---|---|
| `SCRAPED` | merged into `raw_jobs` | `STAGED` |
| `STAGED` | `isRelevantJob` вҶ’ `pass:false` | `REJECTED(reason)` (terminal, logged) |
| `STAGED` | `isRelevantJob` вҶ’ `pass:true` | `PROMOTED` (merged into `jobs`) |
| `PROMOTED` | `isNegativeJob` true | `SCORED(0)` (no Ollama) |
| `PROMOTED` | Ollama returns repairable JSON, not inverted | `SCORED(n)` |
| `PROMOTED` | Ollama unrepairable / inverted / null / timeout | `FALLBACK(-1, review)` |
| `SCORED` / `FALLBACK` | row written | `PERSISTED` (terminal) |

**Key invariant**: every `PROMOTED` job reaches `PERSISTED` вҖ” no job is silently dropped after promotion. `FALLBACK` replaces the current `continue`-without-persist behaviour.
